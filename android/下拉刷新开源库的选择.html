下拉刷新开源库的选择

下拉刷新(Pull-To-Refresh)是App交互中非常常见的一种交互方式，最早是由Facebook在他们的应用中使用。

在Android中，要实现下拉刷新功能，不太可能自己去写，多数情况下要使用开源库。本文将简单介绍三种最常用的下拉刷新类库。

在实际项目中，根据项目需求、规模的不同，对开源库的选择往往也不同，各位可以自己尝试三种实现方式，这样在工作中，就能根据实际情况作出合理的选择，而不是永远只有一种解决方案。

对开源库的选择，可以从以下几个方面来考虑：

  • 稳定性：Bug少，对原生控件代码的侵入尽量少。
  • 可扩展性：例如，要从ListView切换到新控件RecyclerView，此开源库是否支持？
  • 可定制性：下拉刷新的时长、动画、文字、距离等都可自定义，可配置项越多，实现UI需求就越容易；如果为了实现UI效果，需要修改类库的源码，往往会增加很多工作量。
  • 性能影响：有的下拉刷新库效果很炫，但对列表的滑动性能有影响，这在对性能要求高的应用中就不适用。

下拉刷新的常用开源库

目前最流行的下拉刷新实现方式有三种：

开源库Android-PullToRefresh

这是一个比较古老的库，写得很好，以前非常流行，核心代码如下：

资源文件：

<com.handmark.pulltorefresh.library.PullToRefreshListView
    android:id="@+id/pull_to_refresh_listview"
    android:layout_height="fill_parent"
    android:layout_width="fill_parent" />

在Activity或Fragment中：

PullToRefreshListView pullToRefreshView = (PullToRefreshListView) findViewById(R.id.pull_to_refresh_listview);
pullToRefreshView.setOnRefreshListener(new OnRefreshListener<ListView>() {
    @Override
    public void onRefresh(PullToRefreshBase<ListView> refreshView) {
        // 在这里开始获取新数据
        new GetDataTask().execute();
    }
});

private class GetDataTask extends AsyncTask<Void, Void, String[]> {
    ...
    @Override
    protected void onPostExecute(String[] result) {
        // 在这里停止UI的刷新效果
        pullToRefreshView.onRefreshComplete();
        super.onPostExecute(result);
    }
}

从上面的代码能看出，这个开源库的使用还是很简单的。但它在设计上有一个缺陷：我们需要使用PullToRefreshListView来替代ListView，而不是在ListView外层套一个下拉刷新的布局。同样，为了支持常见的几种视图，例如GridView，ScrollView，该项目都提供了一个下拉刷新版本的对应视图，这就造成了它缺乏定制性和扩展性。目前此项目已停止维护。

这个设计上的缺陷是由于，在2013年之前，Android不能很好的支持带资源的类库(不能将XML或图片等资源打包到jar包中)，自定义组件的封装和实现收到了很大的限制。现在，在Android Studio中，我们可以使用aar格式，支持带资源的类库，所以必然会有设计更合理的下拉刷新解决方案。

虽然有这样的缺陷，此开源库技术很成熟，使用广泛，同时支持下拉刷新和上拉刷新，被很多知名App 都使用过，稳定性上不用担心。

另一方面，这种停止维护的类库，用在新版本的Android SDK中，无法保证不出问题，在开发新App时需要慎重选择。

Android官方新控件SwipeRefreshLayout

此控件遵循Google的Material Design风格，它包含在AppCompat包中，使用也很简单，核心代码如下：

资源文件：

<android.support.v4.widget.SwipeRefreshLayout
    android:id="@+id/contentView"
    android:layout_width="match_parent"
    android:layout_height="match_parent">

    <ListView
        android:id="@+id/listView"
        android:layout_width="match_parent"
        android:layout_height="match_parent" />

</android.support.v4.widget.SwipeRefreshLayout>

在Activity或Fragment中：

    swipeRefreshLayout.setOnRefreshListener(new SwipeRefreshLayout.OnRefreshListener() {
        @Override public void onRefresh() {
            // 获取新数据
            ......
        }
    });

    // 刷新结束后调用此方法关闭刷新效果
    swipeRefreshLayout.setRefreshing(false);

作为官方控件，SwipeRefreshLayout使用简便，稳定性高，随着Android SDK的升级始终有新版本的发布，对SDK的适配性最好。

注意到，此时下拉刷新是一个单独的布局，而不再是重写ListView或GridView，你在这个布局中可以放置任意视图，这样，它的可扩展性就明显增强了。

另外，SwipeRefreshLayout遵循官方的Material Design风格，没有提供多种多样的UI刷新样式，这是优点也是缺点：一方面，由于代码精简，控件的性能较好；另一方面，如果你的UI设计师不想遵循Material Design风格，比价有自己的想法，用这个控件可能就难以实现。

开源库android-Ultra-Pull-To-Refresh

这是国内程序员写的一个很优秀的开源库，附有中文文档和介绍，大家可以自行阅读：

中文文档

介绍：我眼中的下拉刷新

依赖：compile 'in.srain.cube:ultra-ptr:1.0.11'

核心代码：

资源文件：

<?xml version="1.0" encoding="utf-8"?>
<in.srain.cube.views.ptr.PtrClassicFrameLayout
    xmlns:tools="http://schemas.android.com/tools"
    android:id="@+id/ptrClassicFrameLayout"
    xmlns:android="http://schemas.android.com/apk/res/android"
    android:layout_width="match_parent"
    android:layout_height="match_parent"
    tools:ignore="HardcodedText">

    <ListView
        android:id="@+id/listView"
        android:layout_width="match_parent"
        android:layout_height="match_parent"/>

</in.srain.cube.views.ptr.PtrClassicFrameLayout>

在Activity或Fragment中：

    ptrFrameLayout.setPtrHandler(new PtrDefaultHandler() {
        @Override public void onRefreshBegin(PtrFrameLayout frame) {
            // 开始更新数据
            updateData();
        }
    });

    // 数据更新完毕，调用下面的方法停止刷新动画
    ptrFrameLayout.refreshComplete();

此类库同样是用一个布局来完成下拉刷新的工作，可扩展性好，能对任何视图做下拉刷新。

同时，它的可定制性也非常强，可以实现各种各样的下拉刷新效果。

当然，由于是个人的开源项目，在稳定性上可能不如官方控件有保证，这就需要使用者自行判断了。

总结

  • 在我们的项目中，推荐大家使用SwipeRefreshLayout或者android-Ultra-Pull-To-Refresh。

  • 上面三个库的设计和编码都非常不错，是自定义控件方面的经典学习材料。

  • 三个库的基本使用方法其实是一样的，都有一个下拉刷新的监听器，和一个停止刷新的方法。这种一致性，一方面是由于它本身就是好的设计；另一方面，类库的设计者也常常会主动将接口设计得同旧类库相似，减小开发者的迁移成本，吸引更多开发者使用。
